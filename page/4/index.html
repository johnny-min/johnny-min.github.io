web后端需要了解的知识

internet是如何工作的？

当我们的计算机发送一个‘你好’的消息给另一个计算机时，大致过程是这样：
1.本地计算机从上到下依次经过application(应用协议层)–TCP(传输协议层)–ip(互联网协议层)–hardware(硬件层)
2.然后经过internet到另一个计算机
3.另一个计算机再从下到上依次经过hardware(硬件层)–ip(互联网协议层)–TCP(传输协议层)–application(应用协议层)

什么是http？

http是超文本传输协议。https是加上了tls(前身是ssl)加密数据，保证了安全性。

浏览器是怎么工作的

对于普通的上网过程，系统其实是这样做的：浏览器本身是一个客户端，当你输入 URL 的时候，首先浏览器会去请求 DNS 服务器，通过 DNS 获取相应的域名对应的 IP，然后通过 IP 地址找到 IP 对应的服务器后，要求建立 TCP 连接，等浏览器发送完 HTTP Request（请求）包后，服务器接收到请求包之后才开始处理请求包，服务器调用自身服务，返回 HTTP Response（响应）包；客户端收到来自服务器的响应后开始渲染这个 Response 包里的主体（body），等收到全部的内容随后断开与该服务器之间的 TCP 连接

flask请求上下文和应用上下文

上下文

在程序中，上下文是指一个个外部变量值的集合

应用上下文

在请求，CLI命令或其他活动期间，应用程序上下文跟踪应用程序级别的数据。而不是将应用程序传递给每个函数，而是访问current_app和 g代理

请求上下文

flask从客户端接受请求时，视图函数如果要处理请的的话，需要访问一些对象，这些对象可能是作为参数传递过来的，也可能是外部变量（如request）。这些个外部变量有特定值才有意义，即上下文。
request就是一个请求上下文，当请求被推送后，request才有意义，才可以使用。

引申

请求上下文：request(记录http请求的相关内容)、session(记录请求会话中的信息)。
应用上下文：g(在同一次请求上下文中设置临时变量属性，供请求上下文使用)、curent_app(保存当前app相关信息)。

生命周期

通常情况下应用上下文和请求上下文具有相同的生命周期。
即：当flask应用开始处理请求时，它将推送应用上下文和请求上下文。当请求结束时，将弹出请求上下文，然后弹出应用上下文。

get和post

get or post

一下仅为个人理解：
1.一般情况下获取数据用get,修改和更新数据用post。
2.get的参数是在url中的，而post参数是在body中携带的。
3.get获取的数据一般是在2KB以内，而post则可以超过2KB。
4.URL太长了不好看。
5.安全性来讲post可能会安全一点，但也到不了哪去。
6.效率上讲get高效一点。
